# Лабораторная работа №1

**Тема**: Разработка онлайн-сервиса Syncro для проведения групповых и персональных конференций

**Цель работы**: Разработать веб-приложение Syncro для проведения групповых и персональных конференций

## Перечень заинтересованных лиц (стейкхолдеров)

- Пользователь — конечный потребитель услуги. Использует приложение для общения с друзьями и коллегами. Заинтересован в удобстве, стабильности связи и богатом функционале.
- DevOps-инженер — отвечает за развертывание, поддержку и бесперебойную работу сервиса. Заинтересован в простом масштабировании, мониторинге и отказоустойчивости системы.
- Разработчик — создает и поддерживает приложение. Заинтересован в четких требованиях, используемых технологиях и поддерживаемой кодовой базе.


## Перечень функциональных требований

Для Пользователя:
1. Пользователь должен иметь возможность зарегистрировать новый аккаунт и войти в существующий.
2. Пользователь должен иметь возможность редактировать свой профиль, включая генерацию аватара с помощью ИИ.
3. Пользователь должен иметь возможность находить, добавлять и удалять других пользователей из списка контактов (друзей).
4. Пользователь должен иметь возможность создавать личные (1 на 1) и групповые чаты, а также отправлять в них текстовые и медиа сообщения (изображения, видео, файлы).
5. Пользователь должен иметь возможность производить голосовые и видео-звонки как в личных чатах, так и в групповых.
6. Пользователь должен иметь возможность транслировать свой экран другим участникам во время конференции.

Для системного администратора:

1. Система должна предоставлять метрики работоспособности ключевых сервисов (серверы, базы данных, медиасервера).

Для разработчика:

1. Изменение / добавление модулей не должно приводить к переписыванию значительной части кодовой базы - архитектура должна обеспечивать гибкость добавления/изменения модулей.
2. Приложение должно корректно выводить статус-коды ошибок, иметь функционал глобальной обработки ошибок.

## Перечень сделанных предположений
1. Система должна обладать механизмами защиты персональных данных пользователя, не светить лишний раз чувствительной информацией(допустим id пользователя, содержание его сообщений).
2.  Система должна обеспечивать уровень безопасности в соответствии с действующим на территории РФ законодательством - ключи шифрования для сообщений не должны превышать 56 бит.
3.  Для предотвращения Mitm в голосовой и видео связи должна присутствовать сигнатура у обоих участников звонка.
## Перечень нефункциональных требований
1. Масштабируемость: увеличение числа пользователей должно сопровождаться корректной работой системы без значимой потери качества связи и увеличения времени отклика
2. Безопасность: чувствительные данные пользователей в централизованном хранилище(РБД) должны быть скрыты/зашифрованы. Данные пользователей в распределенных узлах хранилища должны быть зашифрованы при помощи ключа шифрования. 
3. Удобство сопровождения: при увеличении размеров приложения добавление новых модулей/изменение старых должно встраиваться в архитектуру приложения без затраты значимых временных и трудовых ресурсов - модули системы должны быть атомарны и изолированы от других модулей
4. Производительность: приложение должно обеспечивать быстроту отклика элементов системы
5. Надежность: приложение должно обеспечивать бесперебойную работу основных функций системы за исключением проведения технических, профилактических работ, работ по восстановлению системы после аварийного сбоя, сбоев глобального и регионального интернета.
